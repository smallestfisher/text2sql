# 语义视图脚手架计划

## 1. 当前目标

当前阶段不追求把语义视图做成重型、最终态、已优化的真实数据库对象。

当前阶段的目标是：

- 先搭建语义视图的概念层与字段契约
- 让分类、检索、planner 优先围绕稳定的语义对象工作
- 用 SQL 草案沉淀未来可能落库的实现方向
- 为后续真实数据接入和口径收敛预留清晰落点

这意味着当前更关注“接口是否稳定”，而不是“数据库实现是否已经最终确定”。

## 2. 当前已具备的脚手架

当前仓库里已经有两层脚手架：

- 语义配置层：[semantic/semantic_layer.json](/home/y/llm/new/semantic/semantic_layer.json)
- SQL 草案层：[sql/semantic_view_drafts.sql](/home/y/llm/new/sql/semantic_view_drafts.sql)

其中语义配置层负责：

- 定义语义视图名称
- 定义服务的 domain
- 定义输出字段契约
- 给 retrieval / planner 提供统一分析对象

SQL 草案层负责：

- 沉淀未来真实落库时的实现思路
- 帮助核对底表来源和字段映射
- 帮助后续与真实库结构对齐

## 3. 当前阶段建议做到的程度

当前建议做到以下程度即可：

- 在语义层中维护少量核心语义视图定义
- 为每个语义视图明确 `purpose / serves_domains / source_tables / output_fields`
- 在 SQL 草案中保留可讨论的 view 版本
- 让 query profile 默认优先选择这些语义视图

当前不急着做：

- 围绕语义视图做重型性能优化
- 过早做物化视图/索引治理
- 把测试阶段尚未稳定的口径强行固化

## 4. 当前建议保留的核心语义视图

建议当前只维持少量核心对象：

- `semantic_inventory_view`
- `semantic_plan_actual_view`
- `semantic_demand_perf_view`
- `semantic_demand_unpivot_view`

这四个对象已经足够覆盖库存、计划/实际、需求/销售财务三条主线。

## 5. 后续演进方式

当真实数据、真实问法和稳定口径逐步明确后，再按下面顺序演进：

1. 先修正语义层字段契约与设计说明
2. 再修正 SQL 草案
3. 再决定哪些语义视图真正需要落库
4. 最后再做性能优化和治理收紧

## 6. 判断何时值得真实落库

出现以下信号时，再考虑把草案视图升级为真实数据库对象：

- 某类复杂 SQL 在运行时反复出现
- 某些指标口径已经稳定，不再频繁调整
- 某些高频问法已经比较固定
- 运行时 SQL 复杂度开始明显影响稳定性或可治理性
- 权限和审计开始需要围绕统一分析对象收敛

## 7. 当前结论

当前阶段的正确做法不是“立即把语义视图做重”，而是“把语义视图的架子搭好，并保持后续可调整”。

也就是：

- 先让系统学会围绕语义视图工作
- 暂时接受这些视图仍然是 `draft / logical_scaffold`
- 等真实数据和稳定规则出现后，再把成熟部分落成真实数据库视图
