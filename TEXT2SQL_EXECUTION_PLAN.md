# Text2SQL 系统执行计划

## 1. 文档目标

本文档给出基于 [tables.json](/home/y/llm/new/tables.json) 的 Text2SQL 系统落地执行计划，目标是将系统从“方案设计”推进到“可上线试运行”。

执行计划遵循以下原则：

- 先打基础能力，再做复杂多轮交互
- 先稳定语义层和查询正确性，再优化用户体验
- 先做最小可用闭环，再扩充场景和复杂度
- 以评测集驱动迭代，而不是以规则补丁驱动迭代

## 2. 项目阶段划分

建议将项目拆成五个阶段：

1. 方案冻结与基础建模
2. 查询闭环最小可用版
3. 多轮对话与补问增强版
4. RAG 与示例增强版
5. 试运行与上线优化

## 3. 阶段一：方案冻结与基础建模

### 3.1 目标

完成系统基础设计冻结，明确后续开发约束，避免边开发边改方向。

### 3.2 主要任务

#### 3.2.1 梳理数据语义

基于 `tables.json`，完成语义对象定义：

- `Demand.V`
- `Demand.P`
- `Inventory.Daily`
- `Inventory.OMS`
- `Plan.Daily`
- `Plan.Weekly`
- `Plan.MonthlyApproved`
- `Production.Actual`
- `Sales.Financial`
- `Product.Attributes`
- `Product.Mapping`

#### 3.2.2 产出核心配置

产出以下配置或文档：

- `semantic_model.yaml`
- `join_graph.yaml`
- `action_schema.json`
- `query_draft.schema.json`
- `query_plan.schema.json`
- `conversation_fact.schema.json`

#### 3.2.3 明确业务默认口径

至少明确以下内容：

- 用户说“需求”时默认是否允许使用最新版本
- 用户说“库存”时默认查日库存还是月库存
- 用户说“计划”时是否必须指定计划类型
- `FGCODE` 与 `product_ID` 的跨域映射规则
- 需求宽表的月份展开口径

#### 3.2.4 输出评测样本初版

建立首批评测样本，覆盖：

- 新问题
- 追问
- 补充条件
- 不关联问题
- 无效/不支持问题
- 多表关联问题

### 3.3 交付物

- 系统设计文档
- 执行计划文档
- 语义模型初稿
- join graph 初稿
- 首批评测集

### 3.4 验收标准

- 关键业务对象全部建模完成
- 至少 30 到 50 个真实问法样本可映射到语义对象
- 项目组对默认口径达成一致

## 4. 阶段二：查询闭环最小可用版

### 4.1 目标

先打通单轮查询闭环，不急于一开始就做复杂多轮状态处理。

### 4.2 主要任务

#### 4.2.1 建设语义视图或中间表

优先处理最复杂的需求宽表：

- 构建 `semantic_demand_v`
- 构建 `semantic_demand_p`

同步梳理其他语义对象对应的标准查询视图。

#### 4.2.2 实现 QueryDraft -> QueryPlan -> SQL 编译链路

实现以下能力：

- 语义字段映射
- join graph 读取
- 基础聚合编译
- 时间字段和版本字段处理
- 排序、limit、group by 生成

#### 4.2.3 实现 SQL 安全控制

实现以下校验：

- 只允许 `SELECT`
- 单语句
- 白名单对象校验
- 默认 `LIMIT`
- AST 解析
- 超时与返回条数限制

#### 4.2.4 实现基础聊天接口

提供最小接口：

- `POST /chat/query`
- `GET /chat/sessions/{id}`

### 4.3 交付物

- 单轮查询后端服务
- SQL 编译器初版
- SQL 校验器初版
- 语义视图或中间表脚本

### 4.4 验收标准

- 常见单轮问题可完成查询闭环
- 至少 70% 的基础问法在评测集中可正确编译与执行
- 所有 SQL 都经过安全校验

## 5. 阶段三：多轮对话与补问增强版

### 5.1 目标

在单轮闭环稳定基础上，加入追问、补充问题和上下文重建能力。

### 5.2 主要任务

#### 5.2.1 实现会话事实模型

按轮次保存：

- 原始输入
- `action`
- `QueryDraft`
- `QueryPlan`
- 缺槽位结果
- 补问与回答
- 结果摘要

#### 5.2.2 实现 Action 解释链路

支持以下动作：

- `NEW_QUERY`
- `PATCH_QUERY`
- `ANSWER_CLARIFICATION`
- `CONTINUE_RESULT`
- `ASK_CLARIFICATION`（由编译器缺槽位检查驱动，LLM 仅负责翻译为自然语言补问）
- `NON_DATA_REQUEST`
- `UNSUPPORTED_REQUEST`

#### 5.2.3 实现缺槽位驱动的补问

补问逻辑由编译器驱动，而不是靠流程分支写死。

#### 5.2.4 实现上下文重建

支持：

- 从历史事实中检索最近相关查询
- patch 旧查询
- 新问题自动新建查询上下文
- 不关联问题不错误继承旧条件

### 5.3 交付物

- 会话事实存储模块
- 多轮 query 处理模块
- 补问模块
- 多轮评测集 v2

### 5.4 验收标准

- 对于常见 follow-up 问题，能够较稳定识别为 patch
- 对于明显新问题，能够较稳定识别为 `NEW_QUERY`
- 对于缺关键槽位问题，能够自动触发补问

## 6. 阶段四：RAG 与示例增强版

### 6.1 目标

将系统从“仅依赖 schema 和 prompt”升级为“检索增强的查询规划系统”。

### 6.2 主要任务

#### 6.2.1 建立语义对象索引

索引内容包括：

- 语义对象说明
- 指标定义
- 维度定义
- 同义词
- 常见问法

#### 6.2.2 建立 join 子图索引

为复杂多表场景提供可检索的 join 关系候选。

#### 6.2.3 建立会话事实检索

将历史 `QueryPlan`、用户问题和结果摘要建立检索能力，用于当前轮上下文判断。

#### 6.2.4 建立示例库

示例库至少包含：

- 新问题示例
- 追问示例
- 多表关联示例
- 不支持问题示例
- 补问示例

#### 6.2.5 调整 LLM prompt 与结构化输出

将 LLM 的输入改为：

- 当前问题
- 检索到的语义对象
- 检索到的历史事实
- 检索到的相似示例

输出仍保持 `action + QueryDraft + confidence`。

### 6.3 交付物

- 语义检索索引
- 会话事实检索索引
- 示例库 v1
- RAG 版解释器

### 6.4 验收标准

- 多表场景下的实体识别和 join 选择明显优于无 RAG 方案
- 新问题与追问的区分准确率提升
- 问题理解类错误率下降

## 7. 阶段五：试运行与上线优化

### 7.1 目标

在真实用户和真实问法下收集反馈，修正语义模型、示例库和检索质量。

### 7.2 主要任务

#### 7.2.1 接入用户权限与管理后台

实现：

- 用户登录
- 角色授权
- 查询历史管理
- 示例管理
- 语义模型配置管理

#### 7.2.2 完成监控与审计

记录：

- 输入问题
- 检索结果
- 结构化解释
- SQL 摘要
- 执行耗时
- 补问情况
- 失败类型

#### 7.2.3 运营反馈闭环

建立问题归类机制：

- schema 不支持
- join 错误
- 口径错误
- 补问不合理
- 新问题/追问误判

#### 7.2.4 评测集持续迭代

将线上典型失败样本回灌进评测集与示例库。

### 7.3 交付物

- 可试运行版本
- 管理后台初版
- 审计与监控看板
- 线上问题回灌机制

### 7.4 验收标准

- 支持试点用户稳定使用
- 关键错误类型可被审计和回放
- 线上问题可以归因到检索、解释、编译或语义配置层

## 8. 里程碑建议

建议里程碑如下：

- `M1`：方案冻结，语义模型完成初版
- `M2`：单轮查询闭环完成
- `M3`：多轮对话与补问完成
- `M4`：RAG 与示例增强接入
- `M5`：试运行上线

## 9. 工作拆分建议

### 9.1 数据与语义方向

负责：

- 语义模型
- join graph
- 语义视图
- 指标和维度口径

### 9.2 平台后端方向

负责：

- API
- 会话事实存储
- 编译器
- SQL 执行器
- 权限与审计

### 9.3 算法/模型方向

负责：

- RAG
- 示例库
- prompt
- action / draft 输出质量
- 评测与离线验证

### 9.4 前端方向

负责：

- 查询聊天界面
- 补问交互
- 历史会话
- SQL/口径说明展示
- 管理后台

## 10. 优先级建议

### 10.1 第一优先级

- 语义模型
- 需求宽表标准化
- QueryPlan 与 SQL 编译器
- SQL 安全校验

### 10.2 第二优先级

- 会话事实模型
- 补问机制
- 新问题/追问识别

### 10.3 第三优先级

- RAG
- 示例库
- 管理后台
- 运营能力

原因很明确：如果语义层、编译器和 SQL 安全没打稳，后续把 RAG 加进去也只是在放大不稳定性。

## 11. 风险与应对计划

### 11.1 风险：语义口径迟迟定不下来

应对：

- 在阶段一就冻结业务默认口径
- 需要明确标注“默认值”和“必须补问字段”

### 11.2 风险：多表关联经常出错

应对：

- 严格用 join graph 与 join template
- 复杂跨事实场景先不开放

### 11.3 风险：多轮判断不稳定

应对：

- 采用会话事实检索，不依赖单状态
- 持续扩充 follow-up 示例库

### 11.4 风险：方案复杂，落地慢

应对：

- 严格按阶段推进
- 先做最小闭环，再逐步加增强能力

## 12. 资源建议

最小落地团队建议：

- 后端工程师 2 人
- 算法/LLM 工程师 1 到 2 人
- 前端工程师 1 人
- 数据/业务建模 1 人
- 产品或业务接口人 1 人

如果团队规模更小，建议先以单体服务方式实现，减少早期工程复杂度。

## 13. 验收方法

每个阶段验收都应包含三类检查：

### 13.1 功能验收

- 功能是否按预期跑通
- 接口是否稳定

### 13.2 语义验收

- 查询结果口径是否一致
- 多表场景是否符合业务预期

### 13.3 回归验收

- 既有评测样本是否回归通过
- 新引入能力是否造成历史场景退化

## 14. 最终建议

该项目不适合一开始就追求“全场景、多轮、复杂跨域、零补问”。更可行的执行策略是：

1. 先把语义层和编译器做扎实
2. 再做多轮补问与上下文重建
3. 再引入 RAG 和示例增强
4. 最后通过评测集和真实用户反馈迭代

如果顺序反过来，系统很容易出现“看起来能对话，但底层查询不稳定”的问题，最终维护成本会快速上升。
