# Text2SQL 开发计划

## 1. 目标

基于现有 [tables.json](/home/y/llm/new/tables.json) 与 [readme.txt](/home/y/llm/new/readme.txt)，逐步建设一个面向 Web 用户的持续对话式 Text2SQL 系统。

本计划遵循：

- 弱规则、强语义层
- 先结构化语义，再生成 SQL
- 先支持高频问题，再扩展覆盖面
- 失败优先沉淀到语义层、示例库和视图层，而不是追加 prompt 补丁

## 2.1 横向约束：持续弱化规则

这不是一句原则，而是一条持续执行的开发要求。后续每个阶段都必须优先做下面四件事，而不是直接在代码里补分支：

1. 先看能否把失败场景沉淀到 `semantic_layer.json`
2. 再看能否沉淀到语义视图、示例库、检索特征或 Query Profile
3. 只有在无法配置化表达时，才允许增加少量稳定的基础规则
4. 每次新增规则时，必须同时评估是否可以在后续一轮迁移回语义层

弱化规则的具体落地方向：

- 问题理解：从固定 if/else 迁到 `question_understanding`、`domain_inference`、`extractors`
- 领域规划：从硬编码域分支迁到 `query_profiles`
- SQL 生成：从代码内指标/字段映射迁到语义指标与语义视图
- 权限注入：从固定字段名迁到各域 `permission_scope_fields`
- 检索排序：从固定权重拼接迁到结构化特征与可扩展重排输入

验收要求：

- 新增业务能力时，优先新增配置、示例和语义对象，而不是新增场景补丁
- 代码中的规则应尽量退化为“通用解释器”，而不是“业务 case 集合”

## 2. 阶段划分

### P0：语义基础设施

目标：

- 把输入资料沉淀为结构化语义资产
- 为后续检索、分类、Query Plan、SQL 生成提供统一输入

交付物：

- `semantic/semantic_layer.json`
- `schemas/query_plan.schema.json`
- `examples/nl2sql_examples.template.json`
- 首批语义视图设计草案

验收标准：

- 能覆盖当前表域、核心实体、核心指标、核心关系
- 能表达 `FGCODE / Cell / Array / CF / product_ID` 的阶段语义
- Query Plan schema 可承载单轮和多轮问题

### P1：Query Plan 与 SQL 主链路

目标：

- 建立从用户问题到 Query Plan，再到 SQL 的最小闭环

交付物：

- 语义召回输入格式
- Query Plan 校验器
- SQL 生成器接口
- SQL 语义校验器接口

验收标准：

- 单轮高频问题能生成结构化 Query Plan
- SQL 只使用语义层登记对象
- SQL 可进入执行前校验阶段

### P2：会话与问题分类

目标：

- 支持持续对话和补充问题识别

交付物：

- `session_state` 数据结构
- 问题分类器输入输出协议
- `follow_up / new_related / new_unrelated / invalid / clarification_needed` 判定逻辑

验收标准：

- 能识别补充问题和明显切题/跑题问题
- 上下文继承通过结构化 `context_delta` 实现

### P3：权限与执行

目标：

- 让系统具备真实可执行能力，并保证安全

交付物：

- 用户与角色模型
- 数据权限模型
- SQL 执行器
- 权限注入器

验收标准：

- 使用只读 SQL
- 自动注入数据权限条件
- 具备超时、审计、结果行数限制

### P4：Web 交互与评测闭环

目标：

- 提供可用前端和效果评测手段

交付物：

- Chat API
- 会话 API
- 评测集
- 评测脚本与报告模板

验收标准：

- 支持创建会话、提问、追问、展示 SQL 和结果
- 可以评估意图识别、选表、Join、执行率、结果正确率

## 5. 剩余后端执行计划

下面的计划只针对后端，按“先治理主链路，再增强效果，再补治理闭环”的顺序执行。

### R0：持续弱规则化重构

目标：

- 把仍残留在代码里的业务 case 继续下沉到语义层、示例库和查询 profile

交付物：

- 规则清单与迁移清单
- 新增语义配置项
- 被替换掉的业务硬编码分支

验收标准：

- 新增业务域能力时，不依赖一组不断增长的 Python if/else
- 分类、检索、规划、SQL 生成、权限注入尽量共享同一套语义配置

### R1：SQL 治理与执行约束

目标：

- 把 SQL 校验从字符串级检查提升到结构化校验
- 增加权限过滤完整性校验
- 增加执行约束、超时、复杂度与风险提示

交付物：

- SQL 结构化检查结果
- 权限过滤覆盖校验
- 执行超时/多语句/笛卡尔积风险检查

验收标准：

- SQL 不仅校验来源表，还要校验是否超出 Query Plan 范围
- 对缺失权限条件、缺失时间条件、可疑 Join 给出错误或警告
- 执行层具备基础超时与返回规模治理

### R2：混合检索与示例库扩充

目标：

- 让检索层从“最小样例召回”升级为“结构化召回 + 关键词重排”
- 扩充示例库覆盖高频业务问法和补充问题

交付物：

- 检索打分增强
- 示例管理接口增强
- 首批扩充后的示例模板

验收标准：

- 检索排序能够利用指标、实体、过滤字段、时间粒度、版本语义
- 示例库能覆盖库存、计划实际、需求销售的常见单轮与追问问题

### R3：分类器与 Planner 语义增强

目标：

- 让问题分类更接近“会话状态 + 语义差分 + LLM 判定”的架构设计
- 降低当前分类器对启发式分支的依赖
- 让 LLM 更像基于本地候选与结构化证据的仲裁器，而不是自由重算分类

交付物：

- 更完整的 classification prompt/input
- 结构化语义差分输入
- 更细的澄清策略与会话切换判定
- `context_delta` 字段语义说明、业务化 few-shot、继承目标摘要与仲裁检查清单

验收标准：

- 能更稳定地区分 follow_up / new_related / new_unrelated / invalid
- classification 与 planner 共享统一语义输入
- 在 follow_up 场景下，分类结果不仅输出 `question_type`，还要尽量输出可执行的结构化 `context_delta`

### R4：持久化治理层

目标：

- 将会话、审计、反馈、权限元数据从内存/文件骨架迁移为可持久化结构

交付物：

- Repository 持久化接口
- 会话、审计、反馈、权限元数据表结构
- 后端运行时可切换的存储实现

验收标准：

- 会话与 trace 不依赖进程内存
- 反馈和管理操作可沉淀为后续优化资产

### R5：真实账户与权限体系

目标：

- 从 stub 账户升级到当前 JWT Web 账户体系骨架

交付物：

- 认证抽象接口
- Role / Permission / DataScope 持久化模型
- 管理端权限配置接口骨架

验收标准：

- 后端权限模型不再依赖 stub-login
- 可清晰区分 Web 功能权限与数据权限

### R6：评测与失败回放闭环

目标：

- 为后续效果优化建立评测与回放基础设施

交付物：

- 评测样本格式
- 失败归因字段
- 最小回放脚本骨架

验收标准：

- 可以离线回放典型问题
- 可以记录分类、检索、规划、SQL、执行各阶段失败原因
