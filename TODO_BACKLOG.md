# Text2SQL 后续待办

## 1. 说明

本待办文档基于当前工程与 [TEXT2SQL_ARCHITECTURE.md](/home/y/llm/new/TEXT2SQL_ARCHITECTURE.md) 的差距整理。

当前状态：

- 后端主链路已完成首版
- 语义层、会话、多轮、权限、执行、评测、文件持久化均已有骨架
- 时间抽取已从解析器硬编码迁到 `semantic_layer.json` + `SemanticRuntime`
- 问题分类已接入 `session semantic diff` 和配置化分类规则
- LLM Query Plan hint 已增加域/视图/字段白名单收敛
- 检索已升级为 `example + semantic_view + metric` 的多源召回，并带统一重排
- 向量检索接口骨架已接入，当前默认使用本地轻量向量化占位实现
- 运行时持久化已切到数据库，文件版 `auth/session/audit/feedback` repository 已移除
- 执行层已补统一状态分类，支持 `empty_result / truncated / timeout / db_error` 等返回状态
- LLM 主链路已增加超时、重试和非法 plan/sql 自动回退机制
- 当前更接近“可继续增强的后端 alpha/beta 版”
- 还未达到架构文档中的企业级最终形态

本文按三档优先级整理：

- `必须做`：不做会明显影响系统正确性、可治理性或后续接入
- `应该做`：会明显提升准确率、可维护性和生产可用性
- `可以后置`：对当前后端不是阻塞项，但属于完整平台应补能力

## 2. 必须做

### 2.1 接入真实 LLM 并收紧 Planner / SQL 约束

目标：

- 让当前骨架从“可运行”变成“依赖真实模型的可用链路”

待办：

- 用真实模型替换当前以本地启发式为主的 Planner 补全
- 收紧 Planner 输出字段约束
- 收紧 SQL Generator 对 Query Plan 的字段、表、指标边界控制
- 建立 LLM 异常回退策略和重试边界

原因：

- 当前链路虽然能跑，但效果上限仍受限于启发式逻辑

当前补充进展：

- 已接入真实 LLM client，运行时可使用 `env/.env` 中的模型配置发起真实调用
- 已补 query plan / classification / sql hint 的超时、重试与失败回退
- 当前剩余重点是继续收紧 Planner / SQL 约束与补更多真实链路联调

### 2.2 接入真实数据库执行环境并补执行治理

目标：

- 让执行层具备稳定的真实运行条件

待办：

- 使用真实只读账号执行
- 增加连接失败、超时、慢查询和数据库异常的统一分类
- 增加更明确的执行限制
- 补充对大结果集、空结果、执行失败的统一返回策略

原因：

- 当前执行器已具备骨架，但仍缺少更完整的生产治理

### 2.3 继续弱化规则

目标：

- 把剩余业务补丁继续迁移到语义层、示例库和 Query Profile

待办：

- 继续排查 `question_classifier`、`query_planner`、`sql_generator` 中残余启发式分支
- 把更多字段抽取、澄清策略、视图选择条件迁入 `semantic_layer.json`
- 对新增业务能力坚持“先配置、后代码”的落地顺序

原因：

- 这是当前架构方向的核心约束，不持续做会重新退化成规则堆积

### 2.4 把运行时持久化从文件迁移到数据库

目标：

- 从“文件可持久化”升级到“系统级可治理持久化”

当前状态：

- 已落地 `users / roles / user_roles / data_permissions`
- 已落地 `chat_sessions / chat_messages / session_state_snapshots`
- 已落地 `query_logs / sql_audit_logs / retrieval_logs / feedback_logs`
- 运行时 repository 已改为数据库实现，文件版已移除

剩余增强项：

- 为运行时表补更多管理端查询接口
- 视需要补索引、归档策略和数据保留策略
- 将评测运行记录也迁到数据库，统一运行时治理

当前补充进展：

- 已补 `runtime session/query/retrieval/sql audit` 管理端查询接口
- 已将 `evaluation_runs` 迁入数据库存储，评测运行记录不再落文件

原因：

- 当前已完成首版数据库化，但距离企业级运维治理仍可继续增强

## 3. 应该做

### 3.1 建立真正的混合检索

目标：

- 从当前结构化示例召回，升级为“结构化 + 关键词 + 向量 + 重排”

待办：

- 接入真实 embedding provider / 向量库
- 为检索日志记录命中来源和排序原因
- 继续扩展检索语料，不只覆盖示例，还覆盖更多业务口径说明和字段语义说明

原因：

- 这是后续提升准确率的关键部分

当前补充进展：

- 已将 `example + semantic_view + metric + knowledge(readme/tables)` 纳入统一召回
- 已支持向量检索 provider 配置，当前默认仍兼容本地轻量向量化
- 当前剩余重点是接入稳定的真实 embedding provider / 向量库并补更多业务知识语料

### 3.2 扩充示例库

目标：

- 从当前少量样例扩充到高频业务可覆盖规模

待办：

- 库存场景扩到按工厂、按客户、按型号、按时间粒度
- 计划/实际场景扩到投入、产出、对比、追问
- 需求/销售场景扩到版本、客户、FGCODE、比较分析
- 补更多 follow-up / clarification / unrelated 样例

原因：

- 当前示例库规模还远低于架构文档建议的 `50+`

### 3.3 提升问题分类器

目标：

- 更接近架构文档里的“会话状态 + 语义差分 + LLM 判定”

待办：

- 引入 LLM 参与分类判定，但保持分类结果受结构化语义信号约束
- 对澄清场景加入更细粒度原因标签
- 扩展跨域切题/跑题样例，继续压低误判率

当前补充进展：

- 已接入“结构化规则 + LLM 受限裁决”的分类链路
- LLM 分类仅在多轮/歧义场景参与，且结果受允许问题类型集合约束
- 已补结构化 `reason_code`，便于前端/管理端识别澄清与分类原因

原因：

- 多轮识别能力已有基础，但仍可继续增强稳定性

### 3.4 用成熟 AST 库替换当前轻量 SQL 解析

目标：

- 增强 SQL 校验的可靠性和可解释性

待办：

- 引入正式 SQL AST 解析库
- 支持更准确的字段、函数、子查询、Join、复杂表达式识别
- 把当前正则级检查迁移到 AST 级检查

当前补充进展：

- 已将 SQL 校验器升级为“优先 `sqlglot` AST，缺失时回退轻量解析”的双通道结构
- 当前代码已为正式 AST 接入留好实现路径，待依赖安装后即可启用

原因：

- 当前实现够用，但不是最终形态

### 3.5 完善登录鉴权与基础角色控制

目标：

- 从当前登录骨架升级为稳定可用的 Web 账户与管理端控制

待办：

- 保持 `admin / user` 基础角色控制稳定可用
- 完善用户、会话、反馈、管理端接口的鉴权闭环
- 视需要保留最小结果可见性控制
- 不再扩展复杂 `RBAC/ABAC/组织树` 权限建模

当前补充进展：

- 已补角色查询/维护接口
- 已补独立的数据权限配置接口
- 已补敏感字段可见性控制，支持结果列 `hidden / masked`
- 已支持用户会话列表与会话归档状态管理

原因：

- 当前项目后端重点是登录鉴权和管理能力闭环，不以复杂数据权限治理为目标
